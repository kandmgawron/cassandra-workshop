The Ring & Token Allocation

How data finds its home
  • Murmur3 hashes the partition key → a 64-bit token
  • Each node owns a set of token ranges
  • 16 vnodes per node = balanced distribution without manual allocation
  • Replicas placed on the next N distinct nodes clockwise
nodetool ring — shows the full token map with DC, rack, and load per token
Confidential

Talking points

The key insight: data doesn't live on a specific node by configuration — it lives there because of where its token falls on the ring. Change the partition key and you change which node owns the data.

vnodes (virtual nodes) are the modern way Cassandra handles token allocation. Instead of each physical node owning one contiguous arc of the ring, it owns 16 small, non-contiguous ranges. This makes adding and removing nodes much easier — you're redistributing 16 small ranges instead of one large one.

Fun question to ask the room: "If I change the partition key of a table, where does the data go?" — Answer: it doesn't move. The data is already written. The ring determines where NEW writes go based on the hash of the key at write time.

Replication Factor & Strategy

Every write is sent to RF nodes simultaneously. The coordinator routes the write — it doesn't have to be a replica.
NetworkTopologyStrategy
  • Specify RF per datacenter
  • Places replicas on different racks within each DC
  • Multi-DC aware — handles cross-DC replication
  • Use this in production — always
REPLICATION = {'class':'NetworkTopologyStrategy', 'dc1':3}
SimpleStrategy ⚠️
  • Places RF replicas clockwise with no rack awareness
  • Ignores DCs entirely
  • Dev/test only — never in production
  • Migrating from SimpleStrategy later requires a full rebuild
Lab 1a: nodetool getendpoints sandbox sensor_readings sensor-a — see exactly which 3 nodes own that partition
Confidential

Talking points

With RF=3, every write is sent to 3 nodes before being acknowledged (depending on consistency level). The coordinator handles routing — the client only talks to one node.

NetworkTopologyStrategy is rack-aware: it tries to place replicas on different racks. This is why rack configuration matters so much — we cover that in the Topology section.

The migration cost from SimpleStrategy is high enough that most teams live with it in production rather than fix it. Don't let it get there — use NTS from day one.

Common Q

"Do all 3 replicas need to acknowledge the write?" — No, depends on consistency level. With LOCAL_QUORUM and RF=3, you need 2 acknowledgements. The 3rd node will get the write eventually via anti-entropy, hinted handoff, or read repair.

Lab 1c: We'll stop cassandra-3, prove QUORUM still works, then show ALL fail — live.
Confidential

Talking points

This is where CAP becomes tangible. Cassandra chooses availability over consistency when a node is down — unless you force consistency with ALL.

LOCAL_QUORUM is the sweet spot for 99% of production use cases. It gives you strong consistency (you'll never read stale data after a successful write) while surviving 1 node failure in a 3-node DC.

The "LOCAL_" prefix matters in multi-DC: LOCAL_QUORUM only requires quorum in the LOCAL datacenter. Plain QUORUM requires quorum across ALL DCs — which means cross-DC latency on every read. Never use QUORUM in a multi-DC setup.

Common Q

"What happens to the write that went to the failed node?" — Hinted handoff. The coordinator stores a hint for the write and replays it when the node comes back. Hints are stored for up to 3 hours by default (max_hint_window_in_ms). After that, the node has to be repaired with nodetool repair.

Talking points

With LOCAL_QUORUM and RF=3: the coordinator sends a full read to one replica and a cheap digest (MD5 hash) to a second. If they agree, it returns the result immediately. This is why QUORUM reads don't always hit all replicas — and why they're faster than you might expect.

Lab 1b uses TRACING ON to see this in action. Point out the "Sending READ message" vs "digest read" lines in the trace output.

Read repair happens asynchronously in the background by default (read_repair_chance). If you're seeing frequent read repairs in your trace output, it usually means your nodes are getting out of sync — run nodetool repair to fix the root cause.

Common Q

"Why does my read latency spike sometimes even when all nodes are healthy?" — Often caused by GC pauses on the coordinator or replica. If a node pauses for 400ms for GC, the coordinator waits. This is why the heap settings in the next section matter so much.

04

This section is about the physical layout decisions that are expensive (sometimes impossible) to fix without downtime. Make sure the rack rule lands clearly — it's the most common mistake we see in production clusters.

DCs, Racks & the Snitch

What the snitch does
  • Tells Cassandra the physical topology — DC and rack of every node
  • GossipingPropertyFileSnitch — reads cassandra-rackdc.properties per node, gossips cluster-wide
  • NetworkTopologyStrategy uses this to place replicas on different racks
  • DC maps to: AWS region, GCP region, or logical grouping
  • Rack maps to: AWS AZ, physical rack, failure domain
cassandra-rackdc.properties
# Node in us-east-1a
dc=us-east-1
rack=us-east-1a

# Node in us-east-1b
dc=us-east-1
rack=us-east-1b
Each node has its own copy of this file. The snitch reads it at startup and gossips the info — so the whole cluster knows the layout of every node.
Confidential

Talking points

The snitch is one of those settings that operators set once and forget — until they change it, which is painful. GossipingPropertyFileSnitch is the right choice for all production clusters. It's the only one that correctly handles multi-DC and rack awareness together.

Ec2Snitch and Ec2MultiRegionSnitch exist for AWS but are less flexible. GossipingPropertyFileSnitch works everywhere and lets you define your own DC/rack layout explicitly — much better for controlled environments.

IMPORTANT: If you change the snitch after the cluster is running, you must update every node's rackdc.properties, restart nodes rolling, and then run nodetool rebuild. Changing the DC name especially is dangerous — keyspace replication strategies reference it by name.

The Rule You Can't Break

Number of racks
must be ≥ Replication Factor
With RF=3 you need at least 3 distinct racks
❌ RF=3, 2 racks
One rack holds 2 replicas. If that rack fails, you lose quorum — 2 of 3 replicas gone. QUORUM reads fail even with LOCAL_QUORUM.
✅ RF=3, 3 racks
One replica per rack. Losing any single rack leaves 2 of 3 replicas intact. QUORUM survives.
Fixing rack configuration requires decommissioning nodes, updating rackdc.properties, and re-adding them one at a time. Plan it before you deploy.
Confidential

Talking points

This is the most common topology mistake we see. Teams provision 3 nodes across 2 AZs (2 in us-east-1a, 1 in us-east-1b) thinking "2 AZs should be fine." But with RF=3 and 2 racks, Cassandra MUST put 2 replicas in the same rack — it has no choice.

The consequence: the cluster claims to support QUORUM, but any AZ failure takes out 2 of 3 replicas, not 1. You're running RF=3 with the resilience of RF=1.

The fix is painful: you have to drain and decommission the extra node in the double-stacked AZ, provision a new node in a third AZ, and let it bootstrap. This takes hours and requires a maintenance window in most setups.

Ask the room: "How many distinct AZs are your production nodes spread across?"

Confidential

Talking points

The 50% disk rule catches teams out. They see 70% utilisation and think "still 30% free." But compaction can temporarily double the disk usage of a table — you need headroom for the compaction output before the inputs are deleted. At 70%, a large STCS compaction can easily tip you over 100%.

The 8GB heap rule is firm. The JVM doesn't manage large heaps well — GC pause times scale non-linearly above 8GB. Above 16GB you can see GC pauses of multiple seconds, which the coordinator interprets as node failure and starts rerouting.

The reason you want remaining RAM for page cache: Cassandra reads SSTables from disk, but the OS caches frequently accessed SSTables in RAM. A node with 32GB RAM and 8GB heap has 24GB of page cache — that dramatically reduces disk reads for hot data.

Common Q

"Our nodes have 64GB RAM — should we set heap to 16GB?" — No. Cap at 8GB. Use the extra RAM as page cache. You'll get better read performance from page cache than from a larger heap.

Heap & GC — The #1 Latency Killer

The rules (not guidelines)
  • Max 8 GB — hard ceiling, non-negotiable
  • Xms = Xmx always — prevents resize-induced pauses
  • Young gen: 100 MB × CPU cores (max 800MB)
  • G1GC for JDK 11+ (default since C* 4.0)
  • MaxGCPauseMillis = 300ms
# jvm11-server.options
-Xms8G # must equal Xmx
-Xmx8G
-Xmn800M # young gen
-XX:+UseG1GC
-XX:MaxGCPauseMillis=300
GC pause impact
Max GC < 200ms✅ Healthy
Max GC 200–500ms⚠️ Investigate
Max GC > 500ms🔴 Causing outages
A 500ms GC pause looks like a node failure to the coordinator. It will start rerouting reads — which adds load to the other 2 nodes, potentially triggering their GC.
Confidential

Talking points

The Xms=Xmx rule: if you set Xms lower than Xmx, the JVM starts with a small heap and resizes it as usage grows. The resize itself causes a GC pause. At production scale, a heap resize can pause Cassandra for seconds. Always set them equal.

The reason for the 8GB ceiling isn't ideology — it's G1GC's behaviour. G1GC works by dividing the heap into regions (~2MB each). With a 32GB heap, you have ~16,000 regions. G1's region scanning doesn't scale linearly — you end up with longer and more frequent GC pauses above 8GB than below it. Paradoxically, a 4GB heap can have shorter GC pauses than a 16GB heap.

The cascade effect: node A pauses for GC → coordinator reroutes to B and C → B and C get more load → they GC more frequently → all 3 nodes are in GC loops. This is "GC storm" and it's a common root cause of Cassandra brownouts.

Lab 3a: run nodetool gcstats and look at the Max GC Elapsed column. Anything above 200ms should be investigated.

Compaction Strategies

STCS
SizeTiered
  • Default strategy
  • Groups SSTables of similar size, merges when threshold hit
  • Best for: write-heavy, append-only
  • Watch out: SSTable count grows → read latency spikes
  • Space amplification during compaction
LCS
Leveled
  • Organises SSTables into levels (L0, L1, L2…)
  • Each level is 10× the size of the previous
  • Best for: read-heavy, mixed workloads
  • Watch out: Higher write amplification (more I/O)
  • Predictable read latency — max 1 SSTable per level per read
TWCS
TimeWindow
  • Groups SSTables by time window
  • Windows close → compacted into single SSTable
  • Best for: time-series data with TTL
  • Watch out: Requires time-ordered writes
  • Out-of-order writes break window boundaries
Lab 3b: Create STCS vs LCS tables, insert rows, flush, compare SSTable count and tombstone stats with nodetool tablestats
Confidential

Talking points

Compaction is background I/O that merges SSTables, discards tombstones past gc_grace_seconds, and enforces TTLs. Without compaction, read performance degrades as the number of SSTables grows (more files to check per read).

STCS vs LCS trade-off: STCS has lower write I/O (fewer compactions) but higher read I/O (more SSTables to check). LCS has higher write I/O but very predictable read latency. For tables that are read frequently, LCS is usually worth the extra write cost.

TWCS is designed specifically for time-series data where you write data in time order and then it ages out via TTL. It's very efficient because windows close cleanly — old data gets fully compacted and then dropped when TTL expires. If your writes are out of order (e.g., late data, retries), TWCS can explode — you get windows that never close because they keep getting new data.

You can change compaction strategy on a live table with ALTER TABLE. It takes effect for new compactions — existing SSTables are compacted under the new strategy over time.

CommitLog — Durability vs Performance

Every write hits the commitlog before the memtable. It's the durability guarantee — if a node crashes mid-write, the commitlog replays on restart.
commitlog_sync: periodic
commitlog_sync_period: 10000ms

# 50-70% I/O reduction, ~5% CPU cost
commitlog_compression:
  class_name: LZ4Compressor

# CRITICAL: separate disk
commitlog_directory: /var/lib/cassandra/commitlog
Sync modes
periodic (default): flush every 10s. Up to 10s of writes can be lost on hard crash. Standard for most use cases.

batch: acknowledge after every fsync. Zero data loss on crash, but latency is dominated by disk seek time. Use only when you truly need 0 data loss.
Separate disk is critical. Commitlog writes are sequential — fast on any disk. But if it shares a disk with data files, random data I/O interrupts the sequential commitlog writes → latency spikes on every write.
Confidential

Talking points

The commitlog is a write-ahead log, similar to WAL in PostgreSQL or RocksDB. Every write is appended here first, then applied to the in-memory memtable. When the memtable flushes to an SSTable on disk, the corresponding commitlog segment is deleted.

LZ4 compression: almost always worth enabling. Commitlog writes are CPU-cheap to compress because LZ4 is extremely fast (multi-GB/s), and you reduce I/O by 50-70%. On NVMe this matters less, but on spinning disk or cloud block storage (EBS), the I/O reduction is significant.

The separate disk advice: in production on AWS, this means separate EBS volumes for /var/lib/cassandra/data and /var/lib/cassandra/commitlog. Many teams use one large gp3 volume for everything and wonder why their write latency spikes. The commitlog's sequential access pattern is incompatible with the random access pattern of SSTables on the same device.

Common Q

"We use NVMe SSDs — does the separate disk advice still apply?" — Less critical for NVMe (negligible seek time) but still recommended. The OS scheduler can still create contention between sequential commitlog I/O and random SSTable I/O even on NVMe.

Thread Pool Monitoring — tpstats

nodetool tpstats is your early warning system. By the time clients see errors, these queues have been building for minutes.
Key thread pools
PoolPendingBlocked ReadStage→ reads→ drops MutationStage→ writes→ drops MemtableFlushWriter→ flush lag→ OOM risk CompactionExecutor→ compaction—
Alert thresholds
🔴 ReadStage.Pending > 10
🔴 MutationStage.Pending > 10
🔴 Any pool Blocked > 0
🟡 CompactionExecutor.Pending > 20
Lab 3c: Run cassandra-stress write n=10000 then immediately check tpstats to see active thread counts.
Confidential

Talking points

Cassandra uses a fixed thread pool model — it does not spin up new threads under load. If all threads in ReadStage are busy, incoming reads queue up in Pending. When Pending fills the queue, requests are Blocked (rejected). Blocked = data loss / client errors.

The key insight about timing: by the time you see ReadStage.Blocked > 0, the cluster has been struggling for a while. Pending > 10 is the alert you want to act on — that's when you can still intervene.

MemtableFlushWriter pending is particularly dangerous: if memtable flushes lag behind, memtables grow. Eventually the JVM OOMs. This usually happens when the disk is saturated and flushes are slow — see commitlog section above.

Add tpstats to your monitoring. The metric names in JMX are: org.apache.cassandra.request:ReadStage.PendingTasks and so on. Most Cassandra monitoring agents (DataDog, Prometheus JMX exporter) collect these automatically.

Partition Design — Size Kills Performance

Why partition size matters
  • A partition is the unit of storage, replication, and GC
  • Reads must scan the entire partition from disk
  • Large partitions cause GC pressure (object → heap)
  • Cannot split a partition without schema change
❌ Anti-pattern
PRIMARY KEY (device_id, event_time)
All events for a device in one partition — grows forever.
✅ Bucketed fix
PRIMARY KEY ((device_id, day), event_time)
One partition per device per day — bounded size, easy TTL.
Size thresholds
< 10 MB✅ Healthy
10 – 100 MB⚠️ Monitor
> 100 MB🔴 Problem
> 1 GB🚨 Emergency
Find large partitions: nodetool tablestats | grep "Large partitions"
Confidential

Talking points

The partition is the fundamental unit in Cassandra — it's what gets hashed to determine which node stores it, it's what gets replicated, and it's what gets read from disk as a unit. A 1GB partition means a 1GB disk read — the whole thing gets paged through the JVM, causing massive GC pressure.

The time bucket pattern (device_id + day) is the canonical fix. You need to know your max events per device per day to pick the right bucket size. 10,000 events/day at 200 bytes each = 2MB — fine. 10M events/day = 2GB — too big, use hourly buckets instead.

The query side: once you bucket, queries must always specify the bucket. "Give me all events for device-1 in the last 7 days" becomes 7 queries (one per day bucket), which you run in parallel and merge client-side. This is the Cassandra way.

Lab 4a: insert 1000 rows into a non-bucketed table, flush, and run tablestats. The "Average live cells per slice" will be 1000 — all in one partition. Then create the fixed version and compare.

Tombstones — Deletes That Don't Delete

What happens on DELETE
  1. A tombstone marker is written (not actual deletion)
  2. Tombstones replicate across RF nodes like any write
  3. Tombstones live for gc_grace_seconds (default: 10 days)
  4. Reads must scan through tombstones to find live data
  5. At > 100k tombstones → TombstoneOverwhelmingException
  6. After gc_grace_seconds, compaction can delete the tombstones
❌ DELETE statements at scale
Row-level tombstones accumulate. With 1M deletes/day and 10-day gc_grace, you carry 10M tombstones. Reads scanning them = slow queries + potential failures.
✅ Use TTL instead
Cell-level tombstones from TTL are much smaller and get cleaned up efficiently by TWCS.
default_time_to_live = 604800 -- 7 days
Lab 4b: Delete 1000 rows, flush, run nodetool tablestats — watch tombstone count. Then grep system.log for the tombstone warning.
Confidential

Talking points

The reason tombstones have to live for gc_grace_seconds: if you deleted a row on node A, and node B was down at the time, B doesn't know about the delete yet. If you immediately GC the tombstone on A, and B comes back up later, it will see its copy of the row (without the tombstone) and think the row is live again. gc_grace_seconds gives you a window to repair B before the tombstone can be GC'd.

The 100,000 tombstone threshold (tombstone_failure_threshold): this is a safety valve. When a read encounters more than 100k tombstones in a single partition, Cassandra throws TombstoneOverwhelmingException and aborts the read. This prevents a single bad query from taking down a node. But it means the data is effectively unreadable until you reduce the tombstone count.

Reducing tombstone count on a live cluster: you need to let gc_grace_seconds pass since the last delete, then run nodetool compact to force compaction to run and clear the tombstones. On a busy table, compaction may not run frequently enough — you may need to use nodetool compact explicitly.

Common Q

"Can we lower gc_grace_seconds to speed up tombstone cleanup?" — Yes, but carefully. You must run nodetool repair more frequently than gc_grace_seconds if you lower it. The default 10 days gives you 10 days to repair a failed node before its data diverges permanently. If you lower to 1 day, you must repair daily.

Secondary Index Anti-patterns

❌ Built-in 2i (avoid)
  • Query fans out to every node in the cluster
  • Each node scans its local index
  • Coordinator aggregates results
  • High-cardinality columns → near-full table scan
  • Never on: email, UUID, timestamp, any unique field
CREATE INDEX ON events (user_email); -- ❌
✅ SAI — Storage-Attached Index (C* 4.0+)
  • Node-local index — no cluster-wide scatter
  • Optimised for low-to-medium cardinality
  • Works with ALLOW FILTERING safely
  • Much lower overhead than 2i
CREATE CUSTOM INDEX ON events (status)
USING 'StorageAttachedIndex'; -- ✅
Best pattern: Denormalise into a query table. Write to both tables in the application layer. Reads are always partition-key lookups — no index overhead at all.
Confidential

Talking points

The fan-out problem: when you query a 2i column, Cassandra sends the query to every node and every node scans its local index. For a 30-node cluster, that's 30 parallel scans. If the index is on a high-cardinality field, each scan returns many rows, and the coordinator has to aggregate them all. This is basically a full cluster scan on every query.

When 2i is acceptable: low cardinality columns (status: active/inactive, type: A/B/C) on tables that are queried infrequently. Even then, SAI is better in C* 4.0+.

The denormalisation pattern: this is the "Cassandra way." Design your tables around your query patterns, not around your data model. If you need to query users by email AND by user_id, have two tables: users_by_id and users_by_email. Write to both in a logged batch (for atomicity) or in parallel in the application.

Materialized views: Cassandra has them but they're experimental/unreliable in most versions. Don't use them in production — implement the dual-write pattern in your application instead.
