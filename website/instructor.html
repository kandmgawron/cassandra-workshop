Environment Setup

🎯 Goal: Everyone has a running 3-node cluster ⏱ ~3 min slides · Lab 0 (~10 min) · don't move on until all 3 nodes are UN
🎤 Talking Points
  • The sandbox is intentionally small — 512MB heap per node so it fits on a laptop. Every command, every config, every behaviour is identical to a 30-node production cluster.
  • Two deployment options: Docker (3 containers on one machine, good for most attendees) and AWS CloudFormation (3 real EC2 instances — each node on its own host, genuinely distributed). Ask the room which they're using before starting Lab 0.
  • Docker: While docker/deploy.sh runs, explain what it's doing: pulling the Cassandra 5.0 image, starting 3 containers on a bridge network, waiting for all 3 nodes to reach UN, then loading the demo keyspace and data.
  • AWS: While the CloudFormation stack deploys (~8-10 min), walk through the architecture slides — the stack creates a VPC, security group locked to the deployer's IP, and 3 t4g.small instances with Cassandra pre-configured via user-data. There's no manual setup; the nodes bootstrap automatically and the schema loads on first start.
  • The seed node (the cassandra container on Docker, the first EC2 instance on AWS) is what the other nodes connect to when joining the ring. Once the cluster is up, the seed has no special role — just another peer.

🔬 Deep Dive — How the Bootstrap Works

When cassandra-2 starts, it reads seeds: cassandra from cassandra.yaml and connects to the seed node. The bootstrap sequence:

  1. New node contacts seed, gets cluster name and schema version
  2. New node selects token ranges (16 random vnodes by default)
  3. New node gossips its topology info (DC, rack, tokens) to seed
  4. Seed propagates the new node's info to cassandra-3
  5. Data streaming begins: the 2 existing nodes stream their token ranges to the new node
  6. Once streaming completes, new node moves to UN state

In production, only one node should bootstrap at a time. Our docker-compose uses DependsOn ordering and sleep delays to enforce this — otherwise two nodes racing to bootstrap can corrupt the ring.

The sandbox skips actual data streaming (there's almost no data) — in production with terabytes, bootstrap can take hours.

Section 3 · 30 min

Architecture Refresher

🎯 Goal: Shared vocabulary + CAP theorem made real ⏱ ~12 min slides · Lab 1a (3 min) · Lab 1b (4 min) · Lab 1c (5 min) · 6 min discussion
🎤 Talking Points

The ring & tokens

  • Murmur3 hash of the partition key → 64-bit integer (token). The ring is the range of all possible 64-bit integers arranged in a circle. Each node owns a set of ranges on that circle.
  • vnodes (16 per node) mean each node owns 16 small, non-contiguous arcs instead of one big arc. This makes data distribution more even and makes adding/removing nodes cheaper.
  • Ask: "If I have a sensor-id partition key, and I hash 'sensor-a', I get token X. Which node stores it? The node that owns the token range containing X." It's deterministic — the coordinator doesn't need to look it up.

Consistency levels

  • The key distinction: consistency level is per-operation, not per-cluster. You can read with ONE and write with LOCAL_QUORUM — or vice versa.
  • LOCAL_QUORUM is almost always the right answer for production. Strong consistency, no cross-DC latency penalty, survives 1 node failure.
  • Local_ prefix: In multi-DC, LOCAL_QUORUM only counts nodes in the local DC. Plain QUORUM counts across all DCs — which adds cross-DC network latency to every operation. Never use QUORUM in multi-DC.

Read path

  • With LOCAL_QUORUM: coordinator sends full read to 1 replica, digest read (just an MD5 hash of the data) to a 2nd replica. If they match → return data. If not → read repair: fetch full data from all replicas, reconcile by timestamp, return latest, write corrections back.
  • This is why LOCAL_QUORUM reads are often faster than you'd expect — the coordinator only waits for 2 responses, not 3.

💬 Anticipated Q&A
Q: "What happens to the write that went to the node while it was down?"
A: Hinted handoff. The coordinator stores a "hint" — a record of the write — and replays it to the node when it comes back online. Hints are stored for up to 3 hours by default (max_hint_window_in_ms = 10800000). If the node is down longer than that, you need to run nodetool repair to sync its data from other replicas.
Q: "What's the difference between LOCAL_QUORUM and QUORUM?"
A: QUORUM requires acknowledgements from quorum nodes across ALL datacenters — (sum of RF across all DCs / 2) + 1. In a 2-DC setup with RF=3 in each DC, QUORUM requires 4 acknowledgements, which means at least 1 cross-DC read/write on every operation. LOCAL_QUORUM only requires quorum in the local DC (2 of 3). Use LOCAL_QUORUM in multi-DC, always.
Q: "Can I mix consistency levels — e.g., write at ONE and read at QUORUM?"
A: Yes, and this is a common pattern. Writing at ONE is fast but means the write might not be on all replicas when you try to read. Reading at QUORUM means you'll get the latest data as long as at least 2 replicas have it. This combination is safe as long as your read CL + write CL is greater than RF. ONE + QUORUM (1+2 > 3? No — 3 is not > 3). Safer: ONE write + LOCAL_QUORUM read only works if you can tolerate that the ONE write might not be on a QUORUM replica when you read. For strong consistency: LOCAL_QUORUM write + LOCAL_QUORUM read.
Q: "What is read repair exactly and when does it happen?"
A: Read repair is triggered when a read finds inconsistent data across replicas (digest mismatch). There are two types: synchronous (in the read path — the coordinator fetches all replicas, reconciles, and writes the correct data back before returning to the client) and asynchronous (background repair that randomly checks replicas, configured by read_repair_chance). Synchronous read repair adds latency to the query. You can see it in TRACING output as "Read repair decided to repair replicas".
🔬 Deep Dive — Gossip Protocol Internals

Gossip is a peer-to-peer protocol where each node exchanges state information with 1-3 random nodes every second. The gossip message contains a "generation" (a timestamp from when the node started) and a "heartbeat" (incrementing counter). Each node maintains a table of all other nodes' gossip state.

Key gossip states (from nodetool info and ring membership):

StateMeaning
BOOTNode is bootstrapping — joining the ring, streaming data
NORMAL (UN)Node is healthy, serving traffic
LEAVINGNode is being decommissioned — streaming data away
LEFTNode has fully left the ring
REMOVINGNode is being forcibly removed with nodetool removenode
DOWN (DN)Node not responding to gossip — coordinator won't route to it

A node is marked DOWN when another node fails to gossip with it for phi_convict_threshold intervals (default ~8s). The coordinator immediately stops routing reads/writes to DOWN nodes. Hinted handoff kicks in for writes.

Gossip convergence: in a 3-node cluster, every node knows about every other node within ~1-2 gossip rounds (1-2 seconds). In a 100-node cluster, convergence takes log(N) rounds — still fast, ~7-10 seconds.

Section 4 · 25 min

Cluster Topology & Sizing

🎯 Goal: The rack rule lands, snitch is understood, sizing is practical ⏱ ~10 min slides · Labs 2a–2c (8 min) · 7 min discussion · then 10 min BREAK
🎤 Talking Points

The snitch

  • GossipingPropertyFileSnitch is the right choice for every production cluster. It's the most flexible, works everywhere, and has no downsides compared to alternatives.
  • Point out: each node has its OWN copy of cassandra-rackdc.properties. It's not a cluster-wide setting. The snitch reads it at startup and gossips the info — so every other node learns about this node's DC and rack via gossip.
  • Danger scenario: if you deploy 3 nodes but forget to set cassandra-rackdc.properties on node 3, it defaults to dc1/rack1. Now you have node 3 in the wrong rack, which might cause NetworkTopologyStrategy to put 2 replicas in the same rack — violating your resilience assumptions.

The rack rule

  • This is the one rule where violation is guaranteed to cause production incidents. If racks < RF, one rack holds multiple replicas. If that rack fails, you lose quorum for those partitions.
  • Real story pattern: team provisions 6 nodes across 2 AZs (3 per AZ), sets RF=3. They think they have 6-node RF=3 protection. But with 2 racks, each rack holds 1.5 replicas on average. When us-east-1a fails, they lose 3 nodes, and for partitions that had 2 replicas in that AZ, they drop below quorum.
  • Fixing it in production: decommission one node from the double-stacked AZ, provision a new one in a 3rd AZ, let it bootstrap. Takes hours, requires careful sequencing.

Sizing

  • The 50% disk rule: compaction creates new SSTables before deleting old ones. At peak, you can have 100% overhead on a table being compacted. If you start a major compaction at 70% disk usage, you'll hit 100% before it finishes.
  • The 8GB heap rule: not a guideline, not a starting point. It's a hard ceiling enforced by G1GC's scaling characteristics. Above 8GB you get non-linear GC pause growth.
  • The page cache insight: with 32GB RAM and 8GB heap, you have 24GB for page cache. Frequently accessed SSTables get cached in OS RAM. A hot table that fits in page cache is served at memory speed — much faster than disk. Giving more RAM to the heap steals it from page cache.

💬 Anticipated Q&A
Q: "Can we change the snitch type on a live cluster?"
A: Yes but it's painful. You need to: update cassandra.yaml on every node, update cassandra-rackdc.properties on every node, restart nodes one at a time (rolling restart), then run nodetool rebuild on each node after restart. Changing the DC name also requires updating keyspace replication definitions: ALTER KEYSPACE mykeyspace WITH REPLICATION = {'class': 'NetworkTopologyStrategy', 'new-dc-name': 3};
Q: "What happens if we go above 50% disk and can't add nodes quickly?"
A: Short term: disable major compaction (nodetool disableautocompaction) to stop compaction creating more files. This buys time but makes reads slower. Medium term: clear tombstones by running targeted compaction on specific tables. Longer term: add nodes — when you add a node to the ring, it takes token ranges from other nodes, which means it streams that data off their disks, freeing space.
Q: "How many nodes should we have per DC?"
A: Minimum 3 (to match RF=3 with one node per rack). In practice, size for your write throughput. Cassandra can handle roughly 10-50k writes/sec per node depending on data size and disk type. For reads, it depends heavily on whether your hot data fits in page cache. A good starting point for a new cluster: 3 nodes, monitor tpstats for ReadStage/MutationStage backlog, add nodes when Pending consistently > 5.
Q: "We use cloud instances — what's the recommended instance type?"
A: AWS: i3.2xlarge or i3.4xlarge (local NVMe, 64GB RAM). The local NVMe is the big advantage — no EBS latency. If you must use EBS: gp3 at maximum throughput, separate volumes for data, commitlog, and hints. Avoid io1 — you pay for IOPS you don't need when Cassandra's access pattern is mostly sequential. GCP: n2-standard-32 with local SSDs. Avoid standard persistent disk for data — latency is too unpredictable.
🔬 Deep Dive — vnode Token Allocation

With vnodes (num_tokens = 16 per node), Cassandra allocates token ranges randomly across the ring. This solves a major problem with the old single-token approach:

  • Old approach (single token): You manually assign each node a specific token, dividing the ring into equal arcs. When adding a node, you have to recalculate and reassign tokens — and the new node only takes data from adjacent nodes, creating imbalance.
  • vnode approach: Each node owns 16 small random arcs. When adding a new node, each of its 16 arcs takes data from a different existing node — data is spread across the whole cluster, keeping things balanced automatically.

Trade-off: more vnodes means more SSTables per compaction, slightly higher repair overhead, and more complex nodetool ring output. 16 is the production default and works well for most clusters. At 256 vnodes (used in very large clusters), you can get compaction efficiency issues.

To see actual token assignments: SELECT tokens FROM system.local returns the 16 token start points for the local node.

Section 5 · 40 min

cassandra.yaml Deep Dive

🎯 Goal: Operators leave knowing which knobs matter and why ⏱ ~15 min slides · Lab 3a (4 min) · Lab 3b (8 min) · Lab 3c (5 min, skippable) · 8 min discussion
🎤 Talking Points

Heap & GC

  • Why Xms=Xmx: If you set Xms lower than Xmx, the JVM starts small and resizes the heap as it grows. The resize process triggers a full GC — you get a GC pause at the worst possible moment. Always set them equal.
  • Why the 8GB ceiling: G1GC divides the heap into regions (~2MB each). With 32GB heap = ~16,000 regions. G1 must track every region — metadata overhead grows quadratically. Above 8GB you start seeing diminishing returns on GC pause reduction. Above 16GB, GC pauses can be multiple seconds.
  • The cascade effect: Node A GC pauses → coordinator gives up on it → routes to B and C → B and C get 2x load → they GC more → coordinator gives up on them too → "GC storm." This is one of the most common root causes of Cassandra brownouts. The fix is NOT to increase heap — it's to reduce heap (if over 8GB) or tune GC settings.
  • gcstats interpretation: Max GC Elapsed is the most important column. This is the worst-case GC pause in the last measurement interval. Under 200ms = healthy. 200-500ms = investigate. Over 500ms = causing problems.

Compaction

  • Compaction is background I/O that: merges SSTables, discards tombstones past gc_grace_seconds, enforces TTLs, and reduces SSTable count (which improves read latency).
  • STCS triggers when there are enough SSTables of similar size. As writes come in, new SSTables are created. Eventually 4 SSTables of similar size exist → merge into 1. During merge: disk usage temporarily doubles. This is why 50% max disk is important — compaction needs the headroom.
  • LCS creates SSTable levels (L0, L1, L2…). L1 is 10× the max SSTable size, L2 is 100×, etc. Each level has non-overlapping key ranges, so a read only needs to check 1 SSTable per level. Read latency is much more predictable than STCS.
  • TWCS: only for time-series with TTL. It groups SSTables into fixed time windows (e.g., 1-day buckets). When a window closes (all data in it is older than the window), it's compacted into a single SSTable. When TTL expires on all rows in that SSTable, the whole file is deleted in one shot — no individual tombstone scanning needed.

CommitLog

  • Every write hits the commitlog before the memtable. It's an append-only file — very fast sequential I/O. This is why Cassandra writes are fast even with durability: sequential commitlog writes are much cheaper than random disk I/O.
  • Periodic sync (default): the OS buffers commitlog writes and flushes every 10s. In a power failure, you lose up to 10s of writes. For most applications this is acceptable. For financial transactions, use batch mode (fsync on every write) — but expect 10-50× write latency increase.
  • Separate disk: commitlog writes are sequential; SSTable reads/writes are random. When they share a disk, the disk scheduler interleaves them — sequential writes become effectively random, losing the speed advantage. In production: separate EBS volume or NVMe partition for commitlog.

Thread pools (tpstats)

  • Cassandra uses a fixed thread pool model. ReadStage has 32 threads by default (can be tuned). If all 32 are busy, requests queue (Pending). If the queue fills (usually at 5,120 entries), requests are rejected (Blocked).
  • By the time Blocked > 0, clients are getting errors. The metric you want to alert on is Pending > 10 — that's the queue starting to build, before it overflows.
  • MemtableFlushWriter is often overlooked. If this backs up, memtables can't flush, they grow, and eventually you OOM the JVM. Usually caused by slow disk I/O.

💬 Anticipated Q&A
Q: "How do we change heap settings on a live cluster without downtime?"
A: Rolling restart — one node at a time. The process: 1) Update jvm11-server.options on node 1, 2) nodetool drain to flush memtables and stop accepting writes, 3) Restart cassandra service on that node, 4) Wait for it to return to UN state, 5) Repeat for nodes 2 and 3. With RF=3 and LOCAL_QUORUM, the cluster serves traffic normally throughout — the drained node's load is temporarily absorbed by the other 2.
Q: "How do we pick between STCS, LCS, and TWCS for our tables?"
A: Ask these questions:
1. Is this time-series data with TTL and monotonically increasing timestamps? → TWCS
2. Is this read-heavy (more reads than writes) or do reads need predictable latency? → LCS
3. Is this write-heavy, append-only, and reads are infrequent or can tolerate occasional spikes? → STCS (default)

In practice: most operational tables (audit logs, events, sensor data with TTL) → TWCS. User profiles, lookup tables → LCS. Write-heavy time-series without TTL → STCS. When in doubt, LCS is the safer choice — it just costs more write I/O.
Q: "Our ReadStage.Pending is at 50. What do we do right now?"
A: Immediate triage steps: 1) Check nodetool tpstats on all nodes — is it one node or all of them? One node suggests a local problem (GC, disk). All nodes suggests a cluster-wide issue (overloaded). 2) Check nodetool gcstats for GC storm. 3) Check nodetool tablestats for tables with very high SSTable counts (compaction falling behind). 4) Check disk I/O with iostat. Short-term relief: nodetool compact to force compaction on the worst tables. Longer term: add read capacity (more nodes) or reduce read load (caching, query optimisation).
Q: "Can we use multiple compaction strategies in the same keyspace?"
A: Yes — compaction strategy is per table, not per keyspace. You can and should use different strategies for different tables based on their access pattern. Set it at CREATE TABLE time or change it with ALTER TABLE — the change takes effect for future compactions.
🔬 Deep Dive — G1GC Internals

G1GC (Garbage First Garbage Collector) divides the heap into equal-sized regions (~2MB). Regions are dynamically assigned as Eden (new allocations), Survivor (survived minor GC), or Old (long-lived objects).

Minor GC (Young GC): Runs frequently, only collects Eden + Survivor regions. Fast — typically under 10ms. Objects that survive a minor GC are promoted to Survivor or Old regions.

Mixed GC: G1 initiates when Old region usage exceeds InitiatingHeapOccupancyPercent (IHOP = 70% by default). G1 concurrently marks live objects in Old regions, then collects the regions with the most garbage first (hence "Garbage First"). This is the GC that causes the visible pauses.

Full GC: Happens when Mixed GC can't keep up with allocation rate. The heap is collected all at once — pause scales with heap size. A 4GB Full GC takes ~1s; 8GB takes ~2s; 16GB takes ~5s+. This is the "stop-the-world" event that causes coordinator timeouts.

Tuning for Cassandra: The key settings are MaxGCPauseMillis (target — G1 tries to stay under this) and IHOP (when to start mixed GC). Lowering IHOP (e.g., to 50%) starts GC earlier and keeps pauses shorter but uses more CPU on GC. The -Xmn (young gen size) setting controls how much heap is reserved for Eden — larger young gen means longer intervals between minor GCs but more work per minor GC.

Section 6 · 30 min

Data Modeling Pitfalls

🎯 Goal: They can identify bad patterns in their own schema ⏱ ~10 min slides · Lab 4a (8 min, loop runs in bg) · Lab 4b (5 min) · 7 min discussion
🎤 Talking Points

Partitions

  • The partition is the fundamental unit of Cassandra — it's what gets hashed to a node, replicated to RF nodes, stored contiguously on disk, and read as a unit. A 2GB partition is a 2GB disk operation.
  • The bucketing pattern: instead of (device_id, ts) which puts all events forever into one partition, use ((device_id, day), ts). Now each partition is bounded by one day's events. A device with 1M events/day at 200 bytes each = 200MB partition. Still too big — use hourly buckets instead.
  • The query implication: once you bucket, you must include the bucket in every query. "Get all events for device-1 in the last 7 days" = 7 queries (one per day), run in parallel, merged client-side. This is the Cassandra way — denormalise for your query patterns.
  • Finding large partitions in production: nodetool tablestats | grep "Large partitions" lists partitions larger than the configured threshold (default 100MB). Any entry here needs investigation.

Tombstones

  • The gc_grace_seconds (default 10 days) is a safety window. If you delete a row and a replica is down, the replica doesn't know about the delete. If you GC the tombstone immediately and the replica comes back, it thinks the row is still alive — "zombie data." The 10-day window gives you time to repair the replica before the tombstone disappears.
  • The fix — TTL: instead of explicit DELETE statements, set default_time_to_live on the table. Cassandra writes a small cell-level tombstone when the TTL expires, then cleans it up quickly. This is much more efficient than row-level tombstones from explicit DELETEs.
  • Pattern to avoid: any table where your application regularly runs DELETE statements at scale. If you see "DELETE FROM table WHERE user_id = ? AND ts < ?" in your codebase running thousands of times per minute, you're accumulating tombstones.

Secondary indexes

  • 2i fan-out: every secondary index query hits every node. On a 30-node cluster, that's 30 nodes each doing a local index scan. The coordinator aggregates all results. This is why 2i on high-cardinality columns is catastrophic — each node returns many rows, and the coordinator is processing thousands of results.
  • SAI (Storage-Attached Index, C* 4.0+) is node-local — no fan-out. It's the modern replacement for 2i and is safe for low-to-medium cardinality columns.
  • Best pattern: denormalise into a query table. Write to users_by_id AND users_by_email in the same operation. Reads are always primary-key lookups — zero index overhead.

Lab 4b: tombstones

The tombstone warning in system.log depends on the actual tombstone count hitting the warn threshold (default 1000). If the DELETE command doesn't match exactly, fewer tombstones are created and you might not see the warning. The exact DELETE is:

cql
DELETE FROM device_events_bad
  WHERE device_id = 'device-overloaded'
  AND event_time < toTimestamp(now());
-- This creates a range tombstone (one tombstone covering all timestamps < now)
-- Not 1000 individual row tombstones, but one range tombstone
-- The "1000 tombstones" in the tablestats is because each row must be checked

If system.log is empty (sometimes the Docker log path differs), try:

bash
docker logs cassandra 2>&1 | grep -i tombstone | tail -5
💬 Anticipated Q&A
Q: "We already have large partitions in production. How do we fix them without downtime?"
A: You can't migrate the data automatically — there's no ALTER TABLE that moves data. Your options: 1) Create the new correctly-modelled table alongside the old one. 2) Write a migration job that reads from old table and writes to new table (using your Cassandra driver). 3) Dual-write during transition: application writes to both old and new table. 4) Once new table is caught up, switch reads to new table. 5) Stop writes to old table, eventually drop it. This typically takes days to weeks depending on data volume. Plan it carefully and test in staging first.
Q: "Can we lower tombstone_failure_threshold to allow more tombstones per read?"
A: You can increase it (the default is 100,000) but this is treating the symptom, not the disease. Allowing more tombstones per read means each read takes longer to scan through them. The real fix is reducing tombstone accumulation — either by using TTL instead of DELETE, by running more frequent compactions, or by reducing gc_grace_seconds (with the appropriate repair frequency to match). Raising the threshold buys time but makes read latency worse.
Q: "What's the difference between a row tombstone and a cell tombstone?"
A: A row tombstone is created by DELETE FROM table WHERE pk = ? — it marks the entire row as deleted. A cell tombstone is created by DELETE column FROM table WHERE pk = ? (deleting one column value) or by TTL expiry. Row tombstones are the most expensive because every cell in the row must be considered deleted. Cell tombstones from TTL are more efficient — Cassandra knows exactly which cells to discard without scanning the rest of the row. This is why TTL is much better than explicit row-level DELETEs at scale.
Q: "Is there any use case where built-in 2i (non-SAI) is acceptable?"
A: Yes — low cardinality columns queried infrequently on small tables. Example: a status column (active/inactive) on a user table with 10,000 rows. The fan-out to all nodes is cheap, each node's index scan is tiny, and you might query it only a few times per minute. But in production at scale, the right answer is almost always SAI (C* 4.0+) for indexed columns, or denormalisation into a query table. 2i is generally a code smell in modern Cassandra.
🔬 Deep Dive — SSTable Internals & Compaction Math

An SSTable (Sorted String Table) is an immutable file on disk. Cassandra never modifies existing SSTables — all updates create new SSTables. A "row" in Cassandra might span 10 different SSTables if it's been updated 10 times. Reads must check all SSTables that might contain data for a given partition.

The bloom filter: each SSTable has a bloom filter — a probabilistic data structure that answers "does this partition key exist in this SSTable?" with no false negatives and ~1% false positives. Before reading an SSTable, Cassandra checks the bloom filter. If it says "no", the SSTable is skipped entirely. This is why having many SSTables (STCS) still performs reasonably — most SSTables are skipped via bloom filters.

STCS compaction math: STCS triggers when there are 4+ SSTables of similar size (within a factor of 2). Example: 4 × 100MB SSTables → compacted into 1 × 400MB SSTable. During compaction: disk usage temporarily peaks at 800MB (400MB inputs + 400MB output). After: 400MB. The ratio: compaction requires 1× the size of the output free. With 50% disk: compaction at most produces a file half the disk size → always has space.

LCS compaction math: LCS target SSTable size is configurable (default 160MB). L1 has at most 10 SSTables (total 1.6GB). L2 has at most 100 (16GB). When L1 fills up, the fullest SSTable is compacted with all overlapping L2 SSTables. Write amplification: a write that goes through L0→L1→L2→L3 gets written 4 times. This is the "write amplification" cost of LCS — typically 5-10× the write amplification of STCS.

💬 Common Questions at Session Close
Q: "How do we safely run schema migrations without downtime?"
A: Cassandra schema changes (ALTER TABLE, CREATE TABLE, DROP TABLE) propagate via gossip and don't require restarts — so they're usually safe. However: 1) Never drop a column that's actively being written to — use a two-phase approach (stop writes, then drop). 2) Adding a column is always safe. 3) Changing a column type is not supported — you'd need to create a new column, migrate data, drop old. 4) Changing the primary key structure is not possible without creating a new table and migrating. 5) For large schema migrations, coordinate with rolling application deployments — don't change the schema and deploy a new app version simultaneously.
Q: "When should we add nodes vs resize existing ones?"
A: Add nodes when: disk is the constraint (you need more storage), write throughput is the bottleneck (more nodes = more write capacity), or you need more resilience (more nodes = more failure tolerance). Resize existing nodes when: you're CPU-bound on a few nodes (larger instance), you need more RAM for page cache (larger instance), or you're I/O bound on disk (faster disk on same node). Adding nodes is generally safer than resizing because: you can add one at a time with rolling impact, the new node handles some of the existing nodes' token ranges, reducing load. Resizing requires node replacement (drain → decommission → provision new → bootstrap) — more disruptive.
Q: "How do we diagnose a latency spike in production without making it worse?"
A: Read-only diagnostic commands that don't affect the cluster: nodetool tpstats (thread pool backlog), nodetool gcstats (GC pressure), nodetool tablestats (per-table SSTable count, tombstones), nodetool proxyhistograms (read/write latency percentiles across the cluster), nodetool cfhistograms keyspace.table (per-table latency + row size percentiles). These are all safe to run anytime. Do NOT run nodetool repair or nodetool compact during a live incident — these add I/O and can make things worse. Diagnose first.
Q: "What monitoring do you recommend for Cassandra?"
A: The Cassandra JMX interface exposes hundreds of metrics. The key ones to collect: 1) Thread pools: ReadStage/MutationStage Pending+Blocked (early warning), 2) Latency: ClientRequest.Read.Latency + Write.Latency (p99 and p999), 3) GC: jvm.gc.* (pause duration and frequency), 4) Disk: TableMetrics.*.LiveDiskSpaceUsed per table, 5) Tombstones: TableMetrics.*.TombstoneScannedHistogram. Recommended stacks: Prometheus + JMX Exporter + Grafana (open source, community dashboards available), DataDog (out of box Cassandra integration), or DataStax OpsCenter (commercial). For alerts: pagerduty/opsgenie on ReadStage.Pending > 10 and GC pauses > 500ms.
