Section 3 · 30 min

Architecture Refresher

The ring, gossip, replication, and consistency levels — the model that everything else in Cassandra sits on. We'll cover the theory, then prove it against the live cluster.

The ring & replication

Each node owns a range of token values derived from hashing partition keys with Murmur3. With RF=3, every partition is stored on 3 nodes. NetworkTopologyStrategy ensures replicas are spread across racks within each DC.

🌱
cassandra
seed · tokens 0–33%
🔵
cassandra-2
tokens 33–66%
🔵
cassandra-3
tokens 66–100%
RF=3 → every partition stored on all 3 nodes · gossip keeps topology in sync

Consistency levels

CLNodes requiredWith RF=3
ONE1Fastest, can read stale data
LOCAL_QUORUM2 in local DCStrong consistency, survives 1 node loss ✅
ALL3Max consistency, fails if any replica is down

Lab 1b — Trace a read and see node coordination
4 min

In your cqlsh session, enable tracing. This shows the full lifecycle of a query across the cluster:

cql
TRACING ON;

SELECT * FROM sensor_readings WHERE sensor_id = 'sensor-a';

TRACING OFF;
Look for these lines in the trace
activity                                    | source
------------------------------------------- | -----------
Executing single-partition query            | 172.20.0.2  ← coordinator
Sending READ message to /172.20.0.3         | 172.20.0.2  ← digest read
READ message received                        | 172.20.0.3
Request complete                             | 172.20.0.2
💬
Spot: Which node was the coordinator? How many nodes did it contact? Why did it send a "digest read" to one and a full read to another?

Lab 1c — CAP theorem in action: kill a node, watch consistency fail
5 min

First confirm QUORUM works (requires 2 of 3 nodes):

cql
CONSISTENCY QUORUM;
SELECT * FROM sensor_readings WHERE sensor_id = 'sensor-a';
-- Should return rows ✅

Now stop one node in a separate terminal:

bash — new terminal
docker stop cassandra-3

Back in cqlsh — try both consistency levels:

cql
CONSISTENCY QUORUM;
SELECT * FROM sensor_readings WHERE sensor_id = 'sensor-a';
-- Still works ✅ — 2 of 3 nodes available, quorum = 2

CONSISTENCY ALL;
SELECT * FROM sensor_readings WHERE sensor_id = 'sensor-a';
-- Fails ❌ — NoHostAvailableException, needs all 3

Restart the node and reset consistency:

bash
docker start cassandra-3
# Wait ~30s for it to rejoin, then verify
docker exec cassandra nodetool status
💬
Key insight: This is the CAP trade-off made tangible. QUORUM chose availability over consistency when a node was down. ALL chose consistency and became unavailable. What does your production workload need?
Section 4 · 25 min

Cluster Topology & Sizing

Datacenter and rack layout, snitch configuration, and sizing rules. Getting topology wrong is expensive to fix later — and the sandbox shows you exactly how Cassandra uses this information.

DCs, racks, and the snitch

The snitch tells Cassandra how nodes relate physically. GossipingPropertyFileSnitch (recommended for all production clusters) reads a per-node cassandra-rackdc.properties file and gossips the info cluster-wide.

cassandra-rackdc.properties — one file per node
dc=us-east-1
rack=rack1
prefer_local=true   # use local DC for LOCAL_* consistency levels
⚠️

Rack count must equal RF. With RF=3 and only 2 racks, one rack holds 2 replicas — you can't survive the loss of that rack at QUORUM. Always: racks ≥ RF.

Sizing rules of thumb

Min nodes / DC
3
Matches RF=3, survives 1 node at QUORUM
Max disk usage
50%
Compaction needs headroom — 70%+ causes cascading issues
Heap size
8 GB
Hard ceiling — GC pauses above this can kill nodes
RAM per node
32 GB+
8 GB heap + rest for OS page cache (SSTable reads)

Section 5 · 40 min

cassandra.yaml Deep Dive

The settings operators get wrong most often. We'll look at each one in theory, then observe it live in the cluster — heap, GC, compaction, commitlog, and caches.

Heap & GC

The JVM heap is the single most impactful setting for latency. Too small → constant GC. Too large → stop-the-world pauses that look like node failures.

jvm11-server.options — the rules
# Hard rule: max(¼ RAM, 8 GB) — never exceed 8 GB
-Xms8G
-Xmx8G       # set Xms = Xmx to prevent heap resize pauses
-Xmn800M     # young gen: 100 MB per CPU core (up to 8 cores)

# G1GC — default for C* 4+ / JDK 11+
-XX:+UseG1GC
-XX:MaxGCPauseMillis=300
-XX:InitiatingHeapOccupancyPercent=70
🧪

Lab 3b — Create tables with different compaction strategies and observe
8 min

In cqlsh, create two tables with different strategies, load data, flush to disk, and compare SSTable stats:

cql
USE sandbox;

-- Table 1: default STCS (Cassandra's default)
CREATE TABLE IF NOT EXISTS events_stcs (
  device_id text,
  ts        timestamp,
  reading   double,
  PRIMARY KEY (device_id, ts)
) WITH compaction = {'class': 'SizeTieredCompactionStrategy'};

-- Table 2: LCS for comparison
CREATE TABLE IF NOT EXISTS events_lcs (
  device_id text,
  ts        timestamp,
  reading   double,
  PRIMARY KEY (device_id, ts)
) WITH compaction = {'class': 'LeveledCompactionStrategy', 'sstable_size_in_mb': 10};

-- Insert a row into each
INSERT INTO events_stcs (device_id, ts, reading) VALUES ('dev-1', toTimestamp(now()), 21.5);
INSERT INTO events_lcs  (device_id, ts, reading) VALUES ('dev-1', toTimestamp(now()), 21.5);

Force a flush to disk so we can see SSTable stats:

bash
docker exec cassandra nodetool flush sandbox

# Check SSTable count and partition sizes per table
docker exec cassandra nodetool tablestats sandbox.events_stcs
docker exec cassandra nodetool tablestats sandbox.events_lcs
Key lines to compare in tablestats
SSTable count                  : 1           ← grows over time with STCS
Space used (total)             : 8192
Memtable cell count            : 0
Average live cells per slice   : 1.0
Average tombstones per slice   : 0.0        ← want this near 0
💬
Think about: Which compaction strategy matches your most-queried production tables? Write-heavy logging table → STCS. User profile lookups → LCS. IoT sensor data with 30-day TTL → TWCS.

CommitLog

cassandra.yaml — key settings
commitlog_sync: periodic          # flush every 10s — standard for production
commitlog_sync_period: 10000ms

# LZ4 compression: ~5% CPU overhead, 50–70% I/O reduction

Lab 3c — Check thread pool health with tpstats
5 min

Thread pool stats reveal backpressure and bottlenecks before they become outages:

bash
docker exec cassandra nodetool tpstats
Expected output — key thread pools
Pool Name                   Active  Pending  Completed  Blocked  All time blocked
ReadStage                        0        0       1234        0                 0
MutationStage                    0        0        892        0                 0
CompactionExecutor               0        0         12        0                 0
MemtableFlushWriter              0        0          4        0                 0

# 🔴 Alert if ANY of these are non-zero:
#   ReadStage Pending     → reads backing up (CPU or disk bottleneck)
#   MutationStage Pending → writes backing up
#   MemtableFlushWriter Pending → memtable flush falling behind writes
#   Blocked > 0           → thread pool is full, requests being dropped
bash — generate some load to see active threads
# Run a quick stress test then immediately check tpstats
docker exec cassandra cassandra-stress write n=10000 -rate threads=8 &
sleep 3
docker exec cassandra nodetool tpstats
💬
In production: Set up alerts on ReadStage.Pending > 10 and MutationStage.Blocked > 0. These are early warning signs, not lag indicators — by the time clients see errors, the queues have been building for a while.
Section 6 · 30 min

Data Modeling Pitfalls

Most Cassandra incidents trace back to modeling decisions made months earlier. We'll create each anti-pattern live, observe the symptoms, and fix it.

🧪
Lab 4a — Create a large partition and measure it
6 min

We'll build a table with an unbounded partition — all rows for a given device end up in one partition with no size limit.

cql — create the anti-pattern table
USE sandbox;

-- Anti-pattern: device_id is the partition key, all events pile up here
CREATE TABLE IF NOT EXISTS device_events_bad (
  device_id  text,
  event_time timestamp,
  payload    text,
  PRIMARY KEY (device_id, event_time)
) WITH CLUSTERING ORDER BY (event_time DESC);

Now insert 1,000 rows all into the same partition:

bash — run outside cqlsh
# Insert 1000 rows, all with device_id = 'device-overloaded'
for i in $(seq 1 1000); do
  docker exec cassandra cqlsh -e \
    "INSERT INTO sandbox.device_events_bad (device_id, event_time, payload)
     VALUES ('device-overloaded', toTimestamp(now()), 'data-$i');"
done
echo "Done"

Flush to disk and check partition size:

bash
docker exec cassandra nodetool flush sandbox
docker exec cassandra nodetool tablestats sandbox.device_events_bad \
  | grep -E "SSTable|Partition|cells|tombstone"
Expected output
SSTable count                       : 1
Space used (total)                  : ~120000   ← growing
Partition live row count (estimate) : 1000
Average live cells per slice        : 1000.0   ← all in one partition

Now fix it with a time-bucketed partition key:

cql — the fix
CREATE TABLE IF NOT EXISTS device_events_good (
  device_id  text,
  day        text,         -- '2026-05-01' — caps one day per partition
  event_time timestamp,
  payload    text,
  PRIMARY KEY ((device_id, day), event_time)
) WITH CLUSTERING ORDER BY (event_time DESC)
  AND default_time_to_live = 604800;  -- 7 day TTL

-- Query pattern: always specify both parts of the composite key
SELECT * FROM device_events_good
  WHERE device_id = 'device-1'
  AND   day       = '2026-05-01'
  LIMIT 100;
💬
Rule: A partition > 100 MB is a problem. > 1 GB is an emergency. Find yours with: nodetool tablestats | grep "Large partitions" — anything listed there needs a schema review.
🧪
Lab 4b — Create tombstones and trigger a warning
5 min

Deletes don't remove data — they write a tombstone. Tombstones accumulate until gc_grace_seconds passes (default 10 days). Reads must scan through them.

cql
USE sandbox;

-- Delete most of the rows we just wrote
DELETE FROM device_events_bad
  WHERE device_id = 'device-overloaded'
  AND   event_time < toTimestamp(now());
bash — flush and check tombstone count
docker exec cassandra nodetool flush sandbox
docker exec cassandra nodetool tablestats sandbox.device_events_bad \
  | grep -i tombstone
Expected output
Average tombstones per slice (last five minutes): 1000.0
Maximum tombstones per slice (last five minutes): 1000.0

Now run a SELECT and watch Cassandra warn about tombstones:

cql
TRACING ON;
SELECT * FROM device_events_bad
  WHERE device_id = 'device-overloaded';
TRACING OFF;
bash — check system.log
docker exec cassandra grep -i tombstone /var/log/cassandra/system.log | tail -5
Warning you'll see in system.log
WARN  [ReadStage-...] Read 1000 tombstones from sandbox.device_events_bad;
      query aborted (see tombstone_failure_threshold)

# This is a read that fails with TombstoneOverwhelmingException in production
💬
The fix: Use default_time_to_live on the table instead of explicit DELETEs. TTL tombstones are cell-level and get cleaned up much faster. Pattern to avoid: any table that regularly uses DELETE statements at scale.
🔍
Quick reference — Secondary index misuse

Built-in secondary indexes (2i) fan out to every node on a query. Never use them on high-cardinality columns (email, UUID). Use SAI (Storage-Attached Index, C* 4.0+) for low-cardinality columns, or denormalise into a separate query table.

cql — prefer SAI over 2i
-- Old (avoid for high-cardinality): fans out to all nodes
CREATE INDEX ON events (status);

-- Better (C* 4.0+): node-local index, much more efficient
CREATE CUSTOM INDEX ON events (status)
  USING 'StorageAttachedIndex';
Section 7 · 20 min

Q&A & Homework

Open floor for questions, then a specific homework task so the next session can review your actual cluster — not a generic example.

Key takeaways

  • Use LOCAL_QUORUM in production multi-DC — strong consistency without cross-DC latency
  • Racks ≥ RF — always. Fix this before anything else if it's wrong.
  • Heap hard-capped at 8 GB — the rest goes to OS page cache for SSTable reads
  • Watch tpstats — Pending > 10 on ReadStage or MutationStage is an early warning
  • Bucket partition keys for time-series data — never let a partition grow unbounded
  • Use TTL, not DELETE — tombstones at scale cause read failures

Questions & discussion

Common topics that come up:

  • How to safely change heap settings on a live cluster without a full restart
  • How to pick between STCS / LCS / TWCS for a specific table
  • How to safely run schema migrations without downtime
  • When to add nodes vs. resize existing ones
  • How to diagnose a latency spike in production without making it worse
